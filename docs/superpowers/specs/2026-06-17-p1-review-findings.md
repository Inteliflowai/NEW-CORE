## COMPLETENESS / GAPS — what is missing for an engineer to actually build P1: undefined entities, absent routes/flows, unaddressed locked decisions, and behaviors the spec leaves to chance. Grounded against the actual V1 source at C:/users/inteliflow/core, SCOPE.md, and the mining-findings docs.

- **[HIGH]** The `users` identity table and the 5-role enum are entirely absent from the Data Model (§2), yet the whole auth/RLS/role-routing design depends on them. V1's `supabase/migrations/000_full_schema.sql:42` defines `public.users(id REFERENCES auth.users, school_id, role CHECK IN ('teacher','student','parent','school_admin','platform_admin'), parent_id, grade_level, is_active, ...)`. The spec lists School/Teacher/Class/Student/Profile as separate 'entities' (§2.1) but never says these are ROWS in one `users` table discriminated by `role`, nor where the role string is stored, nor that `auth.getUser()` → role is a lookup against `users.role`. Every per-role layout guard (§1.1, §5.7 'role resolved post-auth'), every RLS policy, and `get_my_school_id()` depend on this table's exact shape (note: V1's enum is `school_admin`/`platform_admin`, but guards.ts also uses `school_sysadmin` — a 6th role the spec's '5 roles' framing never reconciles). An engineer cannot write the first migration or the role guard without it.
  - *Fix:* Add `users` as the canonical identity entity in §2.1: columns `id (FK auth.users)`, `school_id`, `role` (enum — enumerate ALL values incl. `school_sysadmin` or explicitly drop it), `parent_id`, `grade_level`, `is_active`. State that Teacher/Student/Parent/Admin are `users` rows discriminated by `role`, that role is resolved by selecting `users.role` for `auth.getUser().id`, and that this lookup is what the per-role layout guard and `SECURITY DEFINER get_my_school_id()` read. Add the `guardians` table (000_full_schema.sql:62, the parent↔student link) — the spec's Parent screen (§5.3) and `guardStudentAccess` parent check have no data path without it.
- **[HIGH]** Object-level authorization (IDOR protection) is completely unaddressed. The spec's only access-control story is the per-route-group layout guard ('requireRole("teacher")', §1.1) and tier gating (§6). But V1 ships `lib/auth/guards.ts` with `guardClassAccess(classId)` / `guardStudentAccess(studentId)` precisely because — per its own header comment — 'many routes use the service-role admin client which BYPASSES RLS, so the handler itself is the ONLY access control' (introduced by 'Security review 2026-06-16 finding C3' to close cross-class/cross-student IDOR). The spec's `admin.ts` service-role client (§1.4) is used for 'metering, Spark webhook intake, seat triggers' with no mention that every admin-client read of another user's data needs an explicit ownership guard. `/teacher/students/[id]` (§5.1b) is a textbook IDOR target. requireRole alone does NOT stop teacher A from reading teacher B's student.
  - *Fix:* Add a dedicated subsection lifting `lib/auth/guards.ts` (`resolveCaller`, `guardClassAccess`, `guardStudentAccess`, `guardSchoolAdmin`, `guardPlatformAdmin`) and mandate: any route touching the service-role admin client must call the matching object-level guard before reading cross-user data. Explicitly state RLS is NOT a backstop on admin-client paths. Tie `/teacher/students/[id]` and all `[studentId]`/`[classId]` routes to `guardStudentAccess`/`guardClassAccess`.

So per-skill state, the skills registry, AND the cold-start states already exist. Worse, the spec's `computeMasteryBand` 3-band formula (≤50/51-79/≥80, rolling avg of last 5) is the PER-STUDENT quiz band — it is a different instrument from the 6-state per-skill model. The spec conflates them, telling the engineer to 'compute CL per skill via computeMasteryBand' when V1 uses a separate `computeSkillState` against `skill_learning_state`.

Reconcile the two models explicitly: either (a) map the 6 V1 skill states onto the 3 CL verbs (Reinforce/On Track/Enrich) and define that mapping, or (b) state the 3-CL formula replaces the 6-state model and justify discarding `insufficient_data`/`not_attempted`/`needs_more_time` distinctions. Name `skill_learning_state` (not invented `skill_state`) and decide whether to keep `confidence`/`observation_count`/`last_reteach_outcome`.
